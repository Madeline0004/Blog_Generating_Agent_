"Let me quickly walk through the key technologies.

For the backend I'm using FastAPI — it's async, automatically generates API
documentation, and works naturally with Pydantic for data validation.

For embeddings I'm using sentence-transformers with the all-MiniLM-L6-v2 model.
This runs completely locally — no API calls, no cost per embedding. It produces
384-dimensional vectors specifically trained for semantic similarity tasks.

For the vector store I'm using FAISS — that's Facebook's fast similarity search
library. It lets me do cosine similarity search over thousands of document
chunks in milliseconds. The whole index is persisted to disk.

For the LLM I'm using OpenAI's GPT-4o-mini by default, with support for
Anthropic Claude as an alternative. Both are handled by the same interface.

For SEO research I'm using Tavily — it's a search API built specifically for
AI agents. Unlike raw web scraping, it returns clean structured text.

And the frontend is React with Vite, using ReactMarkdown to render the LLM
output as a formatted blog article."

---- 6A. THE RAG PIPELINE ----
[SCREEN: src/utils/chunking.py]

"Let me open the most critical part — the RAG pipeline. I'll start with
chunking.

The assignment specifically asked me to explain chunking decisions — not just
name-drop libraries. So here's what I built.

This function — recursive_character_chunking — works in three levels.

First, it splits the blog text by paragraph boundaries — the double newlines.
Most blog sections fit naturally into paragraphs, so this is the primary split.

If a paragraph is too long — more than 512 tokens — it splits again at sentence
boundaries using regex.

And if a single sentence is somehow too long — it falls back to pure word-count
splitting.

Why 512 tokens? Because 512 tokens is roughly 400 words — enough to capture
a complete argument or explanation, but focused enough that when you retrieve it,
it's actually relevant and not just noisy.

And the overlap — 128 tokens, which is 25% — means when I create the next chunk,
I carry the last 128 tokens from the previous one. This ensures that ideas
spanning a chunk boundary don't get lost."

[SCREEN: src/utils/embeddings.py]

"Now the embedding step. This is the EmbeddingModel class. I've implemented
it as a Singleton — meaning the model loads exactly once per server process,
not on every request. That's important because loading a transformer model
takes a few seconds.

The primary model is all-MiniLM-L6-v2 from sentence-transformers. It outputs
384-dimensional vectors. I also built a hash-based fallback that maintains the
same 384 dimensions using word and character n-gram hashing — so the system
works even if the sentence-transformers library isn't installed."

[SCREEN: src/utils/vector_store.py]

"Now the vector store. I'm using FAISS IndexFlatIP — that's Inner Product.

The key insight here: if I L2-normalise all my vectors to unit length, then
inner product becomes mathematically equivalent to cosine similarity. So I
get cosine similarity scores between 0 and 1 without any extra computation.

This is why I call faiss.normalize_L2() before both adding vectors and before
searching. The index and all metadata are saved to disk, so the next time the
server starts it just loads the existing index instead of re-embedding everything."

[SCREEN: src/services/blog_retrieval.py — find_similar_blog()]

"And the retrieval step. This is find_similar_blog(). It encodes the query
title, searches the FAISS index for the top 10 most similar chunks, and takes
the best score.

If that score is 0.75 or higher, we have a match. We then pull all the chunks
that belong to that matched blog, sort them by their original chunk_index to
reconstruct the correct order, and join them as the full blog content to pass
to the LLM.

If the score is below 0.75, we return None and the orchestrator switches to
the SEO research path."

---- 6B. AGENT ORCHESTRATOR ----
[SCREEN: src/agent/orchestrator.py]

"Now the orchestrator. This is the brain.

The run() method drives everything. It calls _check_library(), checks whether
we got an ExistingBlogMatch back or None, and routes to the right generation path.

What I want to highlight is the error handling. Look at _generate_from_existing()
— if the LLM call fails for any reason, it doesn't crash. It catches the exception,
logs it, and falls back to the SEO research path. Similarly _generate_from_research()
has a final fallback to a mock SEOAnalysis object.

And at the very top level, the whole run() is wrapped in a try/except that
returns a best-effort AgentOutput with the error message. So the API never
returns an unhandled 500 error.

Every single state transition is logged to self.execution_log, which becomes
part of the JSON response and is displayed in the frontend terminal panel."

---- 6C. SEO RESEARCH ----
[SCREEN: src/services/seo_research.py]

"The SEO research service. When Tavily is configured, _real_research() fires.

It searches with depth='advanced' and fetches up to 10 results. For the top 5,
it extracts word counts, heading patterns, and content structure. It then
calculates the average competitor word count, finds common headings using a
frequency Counter, identifies content gaps — topics the competitors missed —
and determines whether the top results favour listicles or comprehensive guides.

All of this becomes the SEOAnalysis Pydantic object that gets passed to the LLM."

---- 6D. LLM GENERATION ----
[SCREEN: src/services/blog_generator.py — show both generate methods]

"The blog generator has two distinct prompt paths.

For the existing blog path, the system prompt tells the LLM: you're an expert
technical writer, study the existing blog's structure and tone, but write a
FRESH post from a different angle — don't duplicate.

For the SEO research path, the system prompt says: you're an SEO-focused writer,
match the competitive word count, fill the identified content gaps.

Context window management: I always truncate the existing blog context to 2000
tokens using the truncate_to_tokens() helper before inserting it into the prompt.
This leaves room for the 4000-token response without hitting the model's limit.

The _format_output() method processes the LLM response: extracts the H1 as the
final title, strips any markdown code-fence wrappers the LLM might add,
extracts the excerpt from the first body paragraph, and runs keyword extraction
using a Counter filtered against a stop-word list."

What I built:
  A full RAG pipeline from scratch — recursive chunking, local embeddings,
  FAISS cosine similarity, threshold-based branching — not just name-dropping

  A state machine orchestrator with documented fallbacks at every node.

  Live web SEO research via Tavily.

  DALL-E 3 image generation with an engineered prompt.

  A React frontend with live agent status updates.

What I'd improve:
  First, incremental indexing — right now you manually re-ingest when new
  blogs are added. I'd add a file-watcher to do this automatically.

  Second, hybrid retrieval — combine FAISS dense search with BM25 keyword
  matching for better recall on exact keyword queries.

  Third, streaming the LLM response to the frontend token by token, so users
  see the blog being written in real time instead of waiting for completion.

The key thing I want to leave you with: every design decision in this project
is justified — the chunk sizes, the similarity threshold, the embedding model,
the choice of Tavily over raw scraping. I can explain and defend every line.
