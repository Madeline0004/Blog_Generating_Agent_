ABSTRACT
--------------------------------------------------------------------------------
This report documents the design and implementation of an autonomous Blog
Generation Agent. The agent accepts a blog title as input and produces a
fully written, SEO-optimised, publication-ready blog post — optionally
including a featured image. The system implements a four-stage pipeline:
semantic search over an existing blog library using FAISS and
sentence-transformers, external SEO research via the Tavily API, LLM-powered
blog generation using OpenAI GPT-4o-mini or Anthropic Claude, and optional
featured image generation via DALL-E 3. A FastAPI backend serves a React
frontend that visualises the agent's state machine in real time. Every
architectural decision — chunk size, similarity threshold, embedding model
choice, fallback strategies — is grounded in engineering reasoning and
documented in the codebase.

--------------------------------------------------------------------------------
PROBLEM STATEMENT (from assignment)
--------------------------------------------------------------------------------
Companies publishing content at scale face two recurring problems:

1. DUPLICATION RISK: When a writer proposes a new blog topic, there is no
   automated way to check whether the company has already covered the same
   topic. Without this check, teams waste effort rewriting similar content,
   and the website accumulates duplicate or overlapping posts that hurt SEO.

2. RESEARCH BOTTLENECK: If the topic IS new, a writer still needs to research
   what content is currently ranking, what headings competitors use, how long
   posts should be, and what angle to take — all before writing a single word.

The assignment requires building an AI agent that automates this entire
workflow: check the existing library → research if needed → generate a
complete, SEO-ready blog post → optionally generate a featured image.

HOW MY PROJECT SOLVES THE PROBLEM
--------------------------------------------------------------------------------
The agent is structured as a state machine with the following flow:

  INPUT: Blog Title
      │
      ▼
  CHECK_LIBRARY ──── Semantic search using FAISS cosine similarity
      │
      ├── score >= 0.75 ──► USE_EXISTING: retrieve full blog content,
      │                     pass as LLM context, generate differentiated post
      │
      └── score < 0.75  ──► SEO_RESEARCH: Tavily web search, competitor
                            analysis, content gaps → pass to LLM
                                │
                                ▼
                          GENERATE: GPT-4o-mini / Claude blog post
                                │
                                ▼
                       GENERATE_IMAGE (optional): DALL-E 3
                                │
                                ▼
                         OUTPUT: .md file + metadata.json + image

Every stage has a documented fallback:
- Library check failure → proceed to SEO research
- SEO API failure → use mock SEOAnalysis with realistic structure
- LLM failure → use high-quality mock blog generator
- Image generation failure → save prompt for manual use

This means the pipeline ALWAYS produces output, even with no API keys,
making it fully evaluable without credentials.

SYSTEM ARCHITECTURE / WORKFLOW
--------------------------------------------------------------------------------

STEP 1 — BLOG LIBRARY INGESTION (RAG Pipeline)
  a) server.py starts → @startup event fires → BlogRetrievalService checks
     if vector store is empty
  b) ingest_blogs() reads all .md/.txt files from data/sample_blogs/
  c) Each file → chunk_blog() → recursive_character_chunking()
       - Level 1: Split by paragraph (double newlines)
       - Level 2: If paragraph > 512 tokens → split by sentence
       - Level 3: If sentence > 512 tokens → split by word count
       - Overlap: 128 tokens carried into next chunk
  d) EmbeddingModel.encode() → all-MiniLM-L6-v2 → 384-dim vectors
  e) faiss.normalize_L2() → unit vectors (enables cosine via inner product)
  f) FAISS IndexFlatIP.add() → vectors stored in memory and on disk

STEP 2 — REQUEST RECEIVED
  a) User enters title in React UI → POST /api/generate
  b) server.py creates AgentOrchestrator and calls run(title, generate_image)

STEP 3 — CHECK_LIBRARY STATE
  a) find_similar_blog(title) encodes title → FAISS search → top-10 chunks
  b) Best cosine similarity score extracted
  c) If score >= 0.75 → ExistingBlogMatch built; all chunks from that blog
     retrieved, sorted by chunk_index, joined as full_content
  d) If score < 0.75 → None returned

STEP 4A — USE_EXISTING STATE (if match found)
  a) full_content truncated to 2000 tokens
  b) System prompt: expert writer, fresh angle, no duplication
  c) User prompt: title + truncated existing blog
  d) LLM call → blog post markdown returned

STEP 4B — SEO_RESEARCH STATE (if no match)
  a) Tavily search: query=title, depth=advanced, max_results=10
  b) Top-5 results analyzed: word counts, headings, competitor structures
  c) Content gaps detected, angle determined
  d) SEOAnalysis object built and passed to LLM

STEP 5 — GENERATE STATE
  a) LLM call with appropriate system/user prompts
  b) Output processed: H1 extracted as title, excerpt from first paragraph,
     keywords extracted via Counter filtering stop-words
  c) BlogPost Pydantic object returned

STEP 6 — GENERATE_IMAGE STATE (optional)
  a) Prompt constructed: "professional illustration of [theme], [keywords]..."
  b) DALL-E 3 API: 1792×1024, standard quality
  c) Image downloaded and saved to data/generated/

STEP 7 — OUTPUT STATE
  a) PublisherService creates timestamped folder in data/generated/
  b) Saves: blog_post.md, raw_content.md, metadata.json, execution_log.txt
  c) AgentOutput Pydantic model returned as JSON to frontend
  d) Frontend renders blog with ReactMarkdown, shows agent logs, enables
     markdown download

KEY CODE EXPLANATION
--------------------------------------------------------------------------------

1. SIMILARITY THRESHOLD — src/config.py
   SIMILARITY_THRESHOLD = 0.75
   This float controls the RAG branch decision. A score of 0.75 on
   cosine similarity (range 0–1) was chosen empirically: low enough to
   catch genuine topic overlaps, high enough to avoid false positives
   where only superficially related terms appear.

2. RECURSIVE CHUNKING — src/utils/chunking.py
   Function: recursive_character_chunking(text, chunk_size=512, overlap=128)
   The algorithm processes paragraphs first. If a paragraph fits within
   512 tokens, it's added to the current chunk. When the chunk is full,
   the last 128 tokens are carried forward as overlap for the next chunk.
   Paragraphs exceeding 512 tokens are split into sentences; sentences
   exceeding 512 tokens are split by word count.

3. EMBEDDING SINGLETON — src/utils/embeddings.py
   class EmbeddingModel uses __new__ for Singleton pattern. The
   SentenceTransformer model is loaded once per process. encode() returns
   L2-normalised 384-dim vectors. The hash-based fallback uses word +
   bigram hashing into the same 384 dimensions, ensuring consistent
   vector store operation regardless of whether the ML library is installed.

4. FAISS INNER PRODUCT — src/utils/vector_store.py
   faiss.IndexFlatIP is chosen over IndexFlatL2. With L2-normalised
   vectors, inner product == cosine similarity. faiss.normalize_L2() is
   called before both add() and search() to ensure unit vectors. Results
   are float scores in [0, 1]. The entire index is persisted to
   data/vector_store/faiss.index and metadata to metadata.pkl via pickle.

5. STATE MACHINE — src/agent/orchestrator.py
   run() is the entry point. It calls _check_library(), inspects the
   return value (ExistingBlogMatch or None), then calls the correct
   generation method. The AgentState Enum defines all states. Every
   internal method appends to self.execution_log, which is included in
   AgentOutput and ultimately rendered in the frontend UI.

6. LLM PROMPT ARCHITECTURE — src/services/blog_generator.py
   Two distinct prompt pairs exist: one for existing blog reference
   (studying tone, avoiding duplication) and one for SEO research
   (matching competitor word count, filling gaps). The existing blog
   context is always truncated to 2000 tokens via truncate_to_tokens()
   before inclusion in the prompt, keeping the total well within the
   4000-token response budget.

7. PYDANTIC MODELS — src/models/schemas.py
   Seven models form the data contract across the entire pipeline:
   BlogChunk → SearchResult → ExistingBlogMatch → SEOAnalysis →
   BlogPost → GeneratedImage → AgentOutput. Every service method is
   type-annotated, and AgentOutput.model_dump() serialises the entire
   pipeline result to JSON for the API response.

LIMITATIONS
--------------------------------------------------------------------------------

1. LLM API Required for Real Generation
   Without an OpenAI or Anthropic key, the mock blog generator activates.
   The mock output demonstrates the pipeline structure but is template-based.

2. Tavily Rate Limits
   Free tier: 1,000 credits/month. Heavy testing can exhaust this.
   The mock SEO research fallback ensures the system still runs.

3. FAISS In-Memory Index
   For very large libraries (thousands of posts), FAISS search may become
   slow without IVF (inverted file) indexing. Production would use FAISS-IVF
   or a managed vector DB (Pinecone, Weaviate).

4. Index Freshness Is Manual
   The index is refreshed only when clear() + ingest_blogs() is called.
   A production deployment would need a file-watcher or CMS webhook.

5. No Authentication on API Endpoint
   /api/generate has no API key or auth middleware. Fine for evaluation,
   not suitable for public deployment without access control.

6. Single-threaded Generation
   One blog is generated per request. Batch generation is not supported.

FUTURE ENHANCEMENTS
--------------------------------------------------------------------------------

1. Incremental Indexing — watch data/sample_blogs/ for file changes and
   update the FAISS index without a full rebuild.

2. Hybrid Search — combine FAISS dense retrieval with BM25 sparse keyword
   matching for better recall on exact keyword queries.

3. CMS Integration — after generation, auto-publish to WordPress, Ghost,
   or Contentful via their APIs.

4. Evaluation Layer — use an LLM-as-judge to score generated blog quality
   against rubrics: SEO score, readability, originality.

5. Multi-step Refinement — generate a draft, run it through a critic prompt,
   then revise. Improves output quality for factual topics.

6. Managed Vector Store — swap VectorStore class for Pinecone/Weaviate for
   production scale with millions of document chunks.

7. Streaming Response — stream the LLM output token-by-token to the frontend
   for immediate visual feedback instead of waiting for the full response.

CONCLUSION
--------------------------------------------------------------------------------

This project demonstrates a complete, end-to-end AI agent system that goes
beyond simply chaining API calls. The Retrieval-Augmented Generation pipeline
is fully implemented from first principles: recursive chunking, sentence-
transformer embeddings, FAISS inner-product similarity with cosine equivalence,
and threshold-based branching. The agent orchestration layer implements a
formal state machine with documented fallbacks at every node. External SEO
research, LLM prompt engineering with context window management, DALL-E 3
image generation, and a polished React frontend complete the system.

Every design decision — the 512/128 chunk parameters, the 0.75 threshold,
the choice of all-MiniLM-L6-v2, the use of Tavily over raw scraping, the
FAISS IndexFlatIP variant — is justified in both the code docstrings and
this report. The system is fully runnable without any API keys, making it
evaluable in any environment.
